# CI/CD as a System

Look at a CI/CD platform with the eyes this site has been training and you'll see a familiar creature: a [queue-and-workers system](../messaging/async-fundamentals.md) (jobs, runners, backpressure), a [DAG scheduler](../case-studies/job-scheduler.md) (stages, dependencies, fan-out), an [artifact store](../data/object-storage.md) with a [content-addressed cache](../networking/cdn.md), and — least appreciated, most important — **the highest-privilege trust boundary in your company**, because the thing that builds and deploys everything can, by definition, touch everything. This page reads the pipeline as architecture, which pays three ways: you operate it better, you can *design one* (a genuinely common Staff-level interview prompt — [full case study here](../case-studies/cicd-platform.md)), and you finally have vocabulary for why builds are slow.

## The pipeline as a distributed system

**Jobs are queue items; runners are competing consumers.** Everything from the [async page](../messaging/async-fundamentals.md) transfers: queue depth and *age* as the health metrics (queue age = developer wait time = the platform's [p99](../foundations/latency-throughput.md)), autoscaling runners on lag ([with the cold-start caveats](kubernetes-autoscaling.md) — warm pools for the latency-sensitive path), priority classes (main-branch and release builds outrank branch pushes; [separate queues beat in-queue priorities](../messaging/async-fundamentals.md)), and [poison jobs](../messaging/async-fundamentals.md) (the build that OOMs its runner forever needs max-attempts and quarantine, not infinite retry).

**Stages form a DAG**, and DAG-think is where pipelines get fast: maximize fan-out (independent test shards in parallel), put the [serial fraction](../foundations/scalability.md) on a diet (the mandatory sequential chain — build → sign → deploy — bounds your best case; everything else should fan out), and fail fast (cheap gates — lint, compile — before expensive ones; [shed doomed work early](../distributed/resilience.md)).

**Caching is the whole performance game.** Three layers, all [content-addressed](../data/object-storage.md): dependency caches (keyed by lockfile hash), [image layer caches](containers.md) (keyed by Dockerfile-layer inputs — order stable-to-volatile or forfeit them), and build/test caches (Bazel-class: keyed by input hashes, skipping *everything* unchanged). The design insight that unifies them: a build cache is a [memoization table over a Merkle tree of inputs](../data/object-storage.md) — hit rates are a function of how honestly you've declared inputs, and "flaky cache" almost always means "undeclared input."

**Hermeticity is what makes caching safe**: builds that depend only on declared inputs (pinned deps, no network fetches of `latest`, no wall-clock/hostname leakage) are reproducible; reproducible builds make cache hits *provably correct* and make "same commit, same artifact" a fact rather than a hope. Every mysterious "works in CI, fails locally" is a hermeticity leak filing a bug report.

## Artifacts: build once, promote everywhere

The invariant that separates mature pipelines: **the artifact is immutable; only its blessing changes.** Build the [image](containers.md) once, at commit; every subsequent stage — staging deploy, integration tests, prod — *promotes the same digest* through environments. Rebuilding per environment reintroduces the exact drift ("staging tested a different binary than prod runs") that the pipeline exists to prevent. Mechanics: digest-pinned references, environment config injected at deploy time ([12-factor separation](deployments.md)), and promotion recorded as metadata (which digest passed which gates when) — that metadata trail *is* your deploy audit log and your [rollback](deployments.md) index.

## The trust boundary nobody draws

Now the security lens, because the pipeline is where an attacker gets the most leverage per compromise: it holds deploy credentials to everything, executes code from every PR, and its output is *trusted by definition* downstream.

- **Untrusted code executes here.** A PR build runs attacker-submittable code on your infrastructure — treat PR runners as hostile: [ephemeral, isolated](containers.md) (fresh micro-VM/container per job, destroyed after — Firecracker-class for public repos), no secrets, no network path to production. The "fork PR steals CI secrets" genre is a solved problem that keeps unsolving itself in configs.
- **Secrets discipline**: deploy credentials live in [short-lived workload identity](secrets-identity.md) (OIDC federation from the CI job to the cloud — no long-lived keys in CI variables, which are the industry's most-breached secret store), scoped per pipeline stage (the test job cannot deploy; the deploy job for service A cannot touch service B).
- **Provenance**: sign what you build ([SLSA-style attestations](containers.md): this digest, from this commit, by this pipeline), and have the cluster [admit only attested images](containers.md) — closing the loop where the pipeline's output becomes the runtime's input. The supply-chain breaches that made headlines (dependency hijacks, build-system compromises) all walked through gaps in exactly this chain.

## Speed as a feature (with numbers)

Pipeline latency compounds into engineering culture: slow pipelines → batched changes → bigger diffs → riskier deploys → more failures → more caution → slower still ([a metastable loop](../distributed/failure-modes.md), organizational edition). The budget worth defending: **PR feedback < 10 minutes** (the context-switch threshold), main-to-production < an hour. The levers, in ROI order: cache honesty (the 20-minute build that's 90% redundant work), test sharding + [flake quarantine](../observability/alerting.md) (a 2% flaky suite at 50 shards fails *most* runs — flakes are a math problem before a discipline problem: quarantine on detection, fix on schedule), DAG fan-out, and warm runner pools. Measure it like production: queue age, stage duration percentiles, cache hit rates, flake rates — [the platform's golden signals](../observability/metrics.md).

!!! ops "DevOps lens"
    The pipeline *is* production — when it's down, nobody ships, and during incidents it's your [mitigation path](../foundations/reliability-availability.md) (the outage where CI is also broken is the outage that lasts all day; that's why the deploy path gets its own [availability tier](../foundations/reliability-availability.md) and a break-glass manual procedure, documented and rehearsed). The operational canon: **runner fleet capacity** (queue age SLO; autoscale with warm minimum; separate pools per trust tier and per priority), **cache infrastructure as tier-0-adjacent** (cache outage = every build reverts to worst case = fleet-wide 10× slowdown that *feels* like an outage), **artifact registry availability** ([already covered](containers.md), doubly true here — it's both pipeline output and deploy input), and **the monorepo-vs-many-repos operational split** (one pipeline config × N repos drift vs. one giant DAG needing selective execution — either way, *pipeline config is code*: reviewed, versioned, canaried like everything else that can take down the fleet).

!!! staff "Staff+ altitude"
    Markers: (1) **The pipeline is the org's metabolism** — deploy frequency, lead time, MTTR, and change-failure rate (the DORA quartet) are *architecture outputs*; a Staff engineer treats "PR-to-prod time" as a system SLO with an owner and a budget, because [every availability claim](../foundations/reliability-availability.md) ultimately routes through how fast and safely you can change things. (2) **Paved-road pipelines**: templates with security (identity, attestation, isolation) and speed (caching, sharding) *built in*, so 40 teams inherit correctness instead of reimplementing it — the pipeline is the paved road's paved road. (3) **The trust boundary is a design review item**: who can modify pipeline definitions (they're deploy-credential-adjacent code), what PR builds can reach, where attestation is enforced — drawn explicitly, audited annually, because the [perimeter moved here](../security/index.md) years ago. (4) **Build-vs-buy with eyes open**: managed CI buys runner ops and sells you queue-position pricing and egress; self-hosted buys control and costs a platform team — the crossover math is [estimation-page](../foundations/estimation.md) material, run honestly per org size.

!!! interview "In the interview"
    "Design a CI/CD system" is a gift of a prompt — answer it with this page's skeleton: *"Queue-and-workers core: jobs queued per trust tier and priority, age-based SLOs, autoscaled ephemeral runners — micro-VM isolation for PR builds, no secrets, OIDC-federated short-lived credentials for deploy stages only. Stages as a DAG: fail-fast gates, parallel test shards, content-addressed caching keyed on declared inputs — hermetic builds make hits provably safe. Build once at commit; promote the signed digest through environments; the cluster admits only attested images. Golden metrics: queue age, stage p95, cache hit rate, flake rate."* That's six systems concepts and two security postures in thirty seconds, and it doubles as the [case-study opening](../case-studies/cicd-platform.md). Probes: *"why are builds slow?"* (cache honesty → serial fraction → shard fan-out, in that diagnostic order); *"how do you handle secrets in CI?"* (OIDC federation, scope-per-stage, hostile PR runners); *"how do you know prod runs what you tested?"* (immutable digest promotion + attestation — the build-once invariant, stated as a chain of custody).
