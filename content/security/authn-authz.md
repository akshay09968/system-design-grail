# Authentication & Authorization

Two questions gate every request in every system you'll ever design: *who are you?* (authentication) and *what may you do?* (authorization). The machinery answering them — sessions, JWTs, OAuth, RBAC — is where interviews probe security most, and where the probing has a favorite trap: the stateless-token story sounds elegant until someone asks how you log a user out. This page is the honest version, trap included.

## Sessions vs. JWTs: the stateful/stateless trade, again

**Server-side sessions** — login creates a record in a [session store](../caching/redis.md); the client holds an opaque cookie ID. Every request looks up the session. Properties: **revocation is trivial** (delete the record — logout, ban, and "sign out all devices" just work), sessions are inspectable and extendable, and the cost is a store lookup per request plus [a stateful tier to operate](../foundations/thinking-in-systems.md) (though a Redis hit is sub-millisecond — the "sessions don't scale" folklore predates modern KV stores and deserves retirement).

**JWTs** — the server signs a token containing the claims (user ID, roles, expiry); any service holding the public key verifies it *locally* — no lookup, no shared store, natural fit for [multi-service fan-outs](../networking/proxies-gateways.md) where the gateway validates once and passes verified identity inward. The price is the trap: **a JWT is valid until it expires, full stop**. Logout, permission downgrade, account compromise — the token doesn't know. Your options, all compromises: short expiry + refresh tokens (the standard: access token 5–15 min, revocable refresh token in secure storage — the dance below; compromise window shrinks to minutes), a denylist checked per-request (congratulations, you've rebuilt the session lookup), or accepting the window (fine for low-stakes claims, unacceptable for [permission-bearing ones](../caching/fundamentals.md)).

The senior synthesis, worth saying verbatim: **JWTs for service-to-service and short-lived access; sessions (or session-backed refresh) where humans log out.** Hybrid is the production norm: opaque session at the edge, JWT minted per-request for the interior hop chain ([the gateway's identity-propagation job](../networking/proxies-gateways.md)).

**The refresh-token dance** — the pattern behind every "stay logged in": short-lived access token + long-lived refresh token; refresh tokens are *stored server-side* (revocable!), single-use with rotation (each refresh issues a new one; reuse of an old one signals theft and kills the whole family — refresh-token-rotation-with-reuse-detection is the phrase that shows you know the current standard). JWT hygiene fine print that separates practitioners: **validate `alg`** (the `none`-algorithm attack is real history), **short `exp` always**, **`aud`/`iss` checks** (a token for service A must not work at service B), and keys rotated via JWKS endpoints ([rotation as engineering](../devops/secrets-identity.md), applied to signing keys).

## OAuth2 & OIDC without the fog

The acronyms unfold into two distinct jobs:

- **OAuth2** is *delegated authorization*: "let app X access my calendar *without* giving it my password." The flow that matters (authorization code + **PKCE**, which is now standard for every client type): app redirects you to the provider → you authenticate *there* and consent → app receives a one-time code → exchanges it (with proof) for tokens. The password never touches the app; the tokens are *scoped* (calendar-read, not everything) and revocable at the provider.
- **OIDC** is *authentication layered on OAuth2*: adds the **ID token** (a JWT asserting who you are) — this is every "Sign in with Google/GitHub" button, and inside companies it's how [SSO](../devops/secrets-identity.md) federates: your services never see passwords, only identity assertions from the IdP. ([Workload identity federation](../devops/secrets-identity.md) — CI proving "I am repo X" for cloud credentials — is this exact machinery with machines as the users; one mental model, humans and workloads both.)

The scoping instinct generalizes everywhere and is a reliable senior signal: tokens carry the *minimum* capability (read:calendar, not \*), short lifetimes, audience-bound — [least privilege](../devops/secrets-identity.md) expressed in token claims.

## Authorization: the harder half

Authentication is a solved-ish protocol problem; authorization is a *modeling* problem that grows with your product:

- **RBAC** — users → roles → permissions. Right default for internal tools and B2B tiers; strains when "editor" needs forty variants ([role explosion](../data/nosql.md) — the cardinality problem, again).
- **ABAC** — policies over attributes ("managers may approve expenses `< $5k` in their own department"). Expressive; auditable with discipline ([policy-as-code](../devops/iac-gitops.md) engines — OPA-class — are ABAC with version control).
- **ReBAC** — permissions as *graph relationships* ("viewer of doc X because member of group Y which is shared on folder Z") — the Google-Zanzibar lineage (SpiceDB-class), and the right model the moment your product has sharing, folders, or org hierarchies. Its system-design honesty: authorization becomes [a graph query service](../data/nosql.md) on every request's critical path — cache it, replicate it, and mind [the staleness-of-permissions trap](../caching/fundamentals.md) (a revoked share that keeps working for 60 seconds is a security bug with a TTL).

Two placement rules that survive every model: **enforce at the resource, hint at the edge** (the gateway can reject the obviously-unauthorized, but the service owning the data makes the real decision — [the fencing-token philosophy](../distributed/coordination.md): the resource participates); and **deny by default** (authorization is an allowlist; every "fail open for convenience" becomes a finding).

!!! ops "DevOps lens"
    The operational surface: **auth outages are total outages** — the [IdP/session store is tier-0](../caching/redis.md) (cache validations with short TTLs so an IdP blip doesn't 401 the world — [fail-static](../devops/kubernetes-architecture.md) for identity, with the security-vs-availability window stated in writing); **clock skew breaks tokens** ([the `iat`/`exp` incidents](../distributed/time-ordering.md) — "not yet valid" errors from one AZ = check NTP before the auth config); **key rotation is a [dual-validity](../devops/secrets-identity.md) dance** (JWKS serves old+new during the roll, or every service 401s in the gap); and **auth logs are your forensics** — login attempts, token issuance, permission denials, [rate-limited](../distributed/rate-limiting.md) credential-stuffing patterns (auth endpoints get the strictest limits and [fail closed](../distributed/rate-limiting.md)). The incident genres: the *mass logout* (session store flush or signing-key mistake — every user, simultaneously, angry), the *ghost permission* (cached authz outliving revocation), and the *federation dependency* ("we can't log in because *their* IdP is down" — know your SSO blast radius).

!!! staff "Staff+ altitude"
    Markers: (1) **One identity platform, not N** — a single IdP/OIDC layer every service trusts, identity propagated via one header convention with verified provenance ([the mesh's mTLS + JWT combo](../devops/service-mesh.md)); five auth schemes is four too many attack surfaces, and consolidation is a Staff-year well spent. (2) **Authorization as a product decision** — the RBAC→ReBAC migration is driven by product shape (sharing! folders!), costs quarters, and is 10× cheaper anticipated than retrofitted; the Staff move is recognizing the product roadmap's authz implications *before* the sharing feature ships on `if owner == user`. (3) **Token design is API design** — claims schemas are [contracts](../networking/apis.md) with all the versioning discipline that implies; the `roles` claim someone stuffed into the JWT in 2024 is load-bearing for forty services now. (4) **Measure the security-latency budget** — authz checks on every request are [latency-budget line items](../foundations/latency-throughput.md); the Zanzibar-style answer (cache + async invalidation + bounded staleness, *documented*) is a worked trade-off, not a hand-wave.

!!! interview "In the interview"
    The set piece is the trap itself — deploy it before they spring it: *"JWTs verified locally at each service for the interior; but revocation is the catch — a JWT is valid until expiry — so: 10-minute access tokens, rotating single-use refresh tokens stored server-side with reuse detection, and the session-revocation semantics live there."* That's the modern standard in one breath. Probes: *"how does Sign-in-with-Google work?"* (OIDC on OAuth2: redirect, authenticate at provider, code + PKCE, exchange for ID token — password never touches you); *"how do you do logout with JWTs?"* (the honest menu: short expiry + revocable refresh; denylist if you must; "you've rebuilt sessions" said knowingly); *"how would you model permissions for a Google-Docs-like product?"* (ReBAC — relationships as a graph, Zanzibar lineage, and the ops honesty: it's a service on the critical path with caching and staleness trade-offs); *"where do you enforce?"* (edge hints, resource decides, deny by default). Security questions in system design are rarely deep — they're checking whether the *defaults* you reach for are the safe ones.
