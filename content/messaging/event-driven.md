# Event-Driven Architecture

There's a moment in every scaling company's life when the integration diagram stops looking like a tree and starts looking like a hairball — every service calling every other service, synchronously, each new feature adding two more arrows. Event-driven architecture is the systemic answer: services stop *commanding* each other ("create an invoice!") and start *announcing facts* ("an order was placed"), and anyone who cares reacts on their own time. Done well, it's the loosest coupling distributed systems offer. Done fashionably, it's the same hairball with extra latency and nobody able to say where an order is. This page is both halves.

## Events are facts, and facts are contracts

An **event** is an immutable statement about the past: `OrderPlaced {order_id, items, total, at}`. Three design decisions hide inside that innocent record, and they're where EDA succeeds or rots:

- **Commands vs. events** — "SendInvoice" (imperative, one intended recipient, coupling by expectation) vs. "OrderPlaced" (fact, zero expectations — billing *happens* to react today; fraud can start reacting tomorrow without the order service knowing). Prefer facts; use explicit commands only where you genuinely mean "do this specific thing, you specifically" ([work queues](async-fundamentals.md)).
- **Fat vs. thin events** — thin ("order 42 changed; call me for details") keeps payloads small but reintroduces synchronous coupling *disguised as async* (every consumer calls back, and the callback is a [availability chain](../foundations/reliability-availability.md) again); fat (the full state snapshot rides along) decouples truly, at schema-discipline prices. The modern default leans fat — *absolute states over deltas*, which [buys idempotency nearly free](delivery-semantics.md).
- **Schemas are public API, forever** — events outlive services; the [contract discipline](../networking/apis.md) (registry, additive evolution, compatibility gates) is *more* critical here than in RPC, because you can't see your callers, and history — [retained in the log](kafka.md) — must stay readable by code not yet written.

## Event sourcing: the log stops being a copy

Conventional systems store *current state* and emit events as exhaust. **Event sourcing** inverts it: the append-only event log **is** the truth; current state is a fold over the log — a cached computation, rebuildable anytime. `balance = replay(deposits, withdrawals)`, literally.

What the inversion buys: a **perfect audit trail** by construction (finance and compliance domains event-source because the ledger *is* the requirement); **time travel** (state as-of any moment — debugging becomes replay, "what did the customer see Tuesday?" becomes a query); **retroactive fixes** (bug in the fold? fix the projection code, replay history, correct state emerges — try that with an UPDATE-mangled table); and **new read models for free** (spin up a projection consuming from event zero — [the offset-rewind superpower](kafka.md), applied to business state).

What it costs, honestly: **schema evolution against immutable history** (you'll read year-old events forever — upcasters and versioned event types are a permanent tax), **snapshotting** (replaying 10M events per entity load is absurd; periodic snapshots + tail-replay is mandatory machinery), **eventual consistency of projections** (reads lag writes — [the whole consistency-models conversation](../foundations/consistency-models.md) returns), and a genuine cognitive/hiring bill. The Staff-grade posture: event-source **the aggregates whose history is the product** (ledgers, orders, anything auditors visit) and let the rest of the system stay boring CRUD. Whole-company event sourcing is almost always fashion exceeding need.

## CQRS: reads and writes stop sharing a model

Command Query Responsibility Segregation formalizes the [oldest asymmetry in this site](../foundations/thinking-in-systems.md): the write side validates commands against invariants (normalized, transactional, small); the read side serves queries from **projections** — denormalized views built by consuming the write side's events, one per query shape ([the NoSQL modeling inversion](../data/nosql.md), arrived at from the other direction: a search index, a feed table, a dashboard aggregate, each fed by the same events).

CQRS pairs naturally with event sourcing but doesn't require it — a plain [CDC feed](../data/analytics.md) off a boring Postgres gives you the same projection machinery with none of the event-sourcing tax, and *that combination (CRUD + CDC + projections) is the pragmatic 80% of CQRS's value at 20% of its cost*. Full CQRS earns its complexity when write contention and read shapes genuinely diverge — feeds, search, analytics-flavored product surfaces — not because a conference talk said models should be segregated.

The bill both flavors share: **the projection lag is user-visible** ("I posted; where is it?") — [read-your-writes patching](../foundations/consistency-models.md) returns as a UI obligation; and **projections are fleet-scale state that must be rebuildable** — the rebuild path (replay at scale, [throttled like any backfill](../data/sql-at-scale.md)) is operational machinery you own from day one, not a someday feature.

## Stream processing: computing on the river

Once facts flow as streams, computation moves *onto* the stream — Kafka Streams/Flink jobs maintaining running state: fraud scores over sliding windows, sessionization, real-time aggregates, [materialized-view maintenance](../data/analytics.md). Two concepts carry 90% of the interview weight:

- **Event time vs. processing time** — when it *happened* vs. when you *saw* it. Mobile clients batch-upload hours late; networks reorder; "revenue per minute" computed on arrival time is fiction during any hiccup. Serious pipelines window by **event time**.
- **Watermarks** — the engine's moving claim that "events older than T have (probably) all arrived," which is what lets a window ever close. Late events (behind the watermark) get a policy: drop, emit corrections, or hold windows open longer — **completeness vs. latency is the fundamental dial**, and there is no setting called "both." Downstream consumers must tolerate *revisions* — which is why streaming outputs are best modeled as upserts, [absolute states again](delivery-semantics.md).

Plus the operational truth: stateful stream jobs are **stateful services** — their internal state (windows, counts, joins in flight) needs checkpointing, restore-on-failure, and rescaling stories ([Flink checkpoints, Streams changelogs](kafka.md)); "it's just a consumer" undersells what you've deployed by roughly one distributed database.

## The honest failure mode: the distributed hairball

The EDA pathology worth naming before an interviewer does: **choreography sprawl** — thirty services reacting to each other's events, no one owning the end-to-end flow, debugging by log archaeology across twelve consumers ("where is order 84712 *right now*?" has no answerable place). The antidotes: [orchestration for flows that need an owner](../data/distributed-transactions.md) (sagas with state machines for money paths; choreography for genuinely independent reactions), **event catalogs + lineage** (who publishes what, who consumes what — discoverable, not tribal), **correlation IDs** threaded through every event ([tracing's](../observability/tracing.md) async cousin), and end-to-end SLOs per business flow so "the order pipeline is slow" is a measured fact, not a vibe. EDA removes coupling of *availability*; it does nothing for coupling of *understanding* — that you must build deliberately.

!!! ops "DevOps lens"
    Operating EDA means operating *history*: **replay is your superpower and your loaded gun** — rebuilding a projection means one consumer reprocessing months of events at full speed, which will [evict page cache](kafka.md), saturate the projection's database, and starve live traffic unless throttled and capacity-isolated (replays are backfills; treat them with backfill ceremony). **Consumer lag per projection is user-visible staleness** — dashboard it as such ("search index: 4 s behind; feed: 800 ms"), alert on the product SLO, not the raw number. **Poison events in choreography** halt reactions silently ([DLQ discipline](async-fundamentals.md), plus alerting per consumer, because one team's stalled consumer is another team's missing side effect). And **watermark/late-data metrics** on streaming jobs (lateness histograms, dropped-event counts, correction rates) are the difference between "the dashboard is wrong" mysteries and a tuning conversation.

!!! staff "Staff+ altitude"
    Markers: (1) **Adopt EDA per seam, not per company** — the inventory between "which flows are genuinely multi-reactor facts" (order lifecycle: yes) and "which are commands wearing event costumes" (SendInvoice with one consumer: just call it or queue it) prevents the hairball at the architecture review, which is the only place it can be prevented cheaply. (2) **Event catalog + schema governance before consumer #3** — the registry, ownership map, and compatibility gates are the difference between an integration fabric and archaeology; retrofitting them onto 200 undocumented topics is a year of Staff-level penance (ask anyone who's done it). (3) **Event-source where history is the product, CRUD+CDC elsewhere** — and write the criteria down, because "should we event-source this?" recurs quarterly forever. (4) **Own the end-to-end flows explicitly** — every business-critical path gets a named owner, an orchestrator or a documented choreography, a correlation-ID convention, and a flow-level SLO; "the events will sort themselves out" is how EDA earns its bad reputation.

!!! interview "In the interview"
    Deploy the vocabulary precisely and sparingly: *"Order placement publishes an `OrderPlaced` fact — fat payload, schema-registered; billing, fraud, and notifications each consume independently, idempotent by event ID. The order *flow* itself is orchestrated — [saga with a state machine](../data/distributed-transactions.md) — because money paths need an owner; the reactions are choreographed because they're genuinely independent."* That sentence shows you know both modes *and* when each applies. Probes: *"why events over API calls?"* (temporal decoupling + unknown future consumers + the availability-chain math — then concede the trade: eventual consistency and flow opacity, with the antidotes named); *"what's event sourcing good for?"* (audit-grade domains + time travel + projection rebuilds; then the honest tax list — schema-forever, snapshots, projection lag — and the "one aggregate, not the whole company" posture); *"how do you handle late events?"* (event time + watermarks + a stated completeness-vs-latency choice + corrections as upserts). The differentiating instinct throughout: EDA is a *coupling* decision, not a technology purchase.
