    Typeahead¶

    Design search autocomplete: suggestions appearing under a search box within a keystroke's patience. The prompt looks small and is secretly a masterclass in read-path optimization — the strictest latency budget in the canon (double-digit milliseconds, per keystroke, at massive QPS), solved by precomputing almost everything and caching the rest. If the feed is the write-time-vs-read-time trade at product scale, typeahead is the same trade compressed into one endpoint.

    Requirements & estimation¶

    Scope: prefix → top-K suggestions (K≈10), ranked by popularity, updated as trends shift (minutes-fresh, not seconds — ask, because it halves the design); personalization and spell-tolerance parked as layers. Non-functional: p99 < 50 ms server-side (the box feels broken beyond ~100 ms end-to-end), and the QPS multiplier that defines the problem: every keystroke is a query — a 10-character search = 10 requests.

    Numbers: 100M searches/day × ~8 keystrokes ≈ 8×10⁸ lookups/day ≈ ~10k QPS average, 30k+ peak — read-QPS on par with the main search engine itself, for a decorative feature. Corpus: top 10M queries; storing top-10 per prefix for prefixes up to length ~8 ≈ tens of GB — fits in RAM across a small cluster. Verdict: "this is a pure read-serving problem over a smallish precomputed dataset — the design is the data structure, the caching, and the pipeline that refreshes it."

    The core structure: precomputed top-K per prefix¶

    The naive answer (trie walked at query time, collecting and ranking descendants) dies on the budget: a popular prefix like "s" has millions of descendants — you cannot rank them per keystroke. The fix is the prompt's whole lesson, the precompute lever at maximum: store the answer, not the data — for every prefix, the top-K suggestions, already ranked:

    • Structure: a hash map prefix → [top-K] is honestly sufficient (and easier to shard than a trie — say this; the trie-vs-map pragmatism reads as experience, not ignorance). A trie with top-K cached at each node is the classic answer and fine too; the map wins on operational simplicity, partitioning (hash the prefix), and rebuild simplicity.
    • Serving: request "sys" → one hash lookup → return 10 strings. Memory-speed, O(1), no ranking at read time — the 50 ms budget closes with ~45 ms to spare, which is exactly the headroom the network path needs.
    • Sharding: by prefix hash; hot prefixes (every user types "t") get the L1-cache-on-serving-nodes treatment — short-prefix results are so hot and so small that every edge node can hold them.

    Edge caching goes further here than anywhere: responses are tiny, shared across all users (pre-personalization), and keyed by prefix — CDN-cacheable with 60 s TTLs at hit rates that beggar belief, because prefix popularity is Zipf on steroids. A double-digit percentage of the fleet's QPS can terminate at the edge. (Client-side debounce — fire after ~100 ms of typing pause — plus client caching of just-typed prefixes cuts another large slice; the client is part of the read path, as ever.)

    The refresh pipeline: where writes went¶

    Nothing updates at query time; popularity flows through a batch/streaming pipeline — the CQRS shape, unusually pure (the serving store is entirely a projection):

    Search logs → stream aggregation (query counts, sliding windows with decay — yesterday's hit shouldn't outrank this hour's trend; exponential decay is the one-line answer) → periodic top-K computation per prefix → build the new serving snapshot → atomic swap.

    The swap is the operational elegance worth narrating: build the next generation of the prefix map offline, verify it (sanity checks: size deltas, canary queries, the blocklist below), then flip serving to the new snapshot — versioned, instantly rollbackable, no in-place mutation of the hot structure ever (immutable-snapshot serving — the same pattern as search-index deploys and feed-model rollouts; naming the family is the flourish). Trend freshness = pipeline cadence (5–15 min batch, or a small streaming lane layered on top for breakout queries — the hybrid, if the interviewer pushes freshness).

    The filter nobody remembers until production: suggestions are user-generated content that your product actively recommends — profanity, harassment targets, legal removals, and spam campaigns (bots typing a phrase to make it "trend") all need a blocklist + anomaly detection in the pipeline, applied at build time (cheap, auditable) with an emergency path that can purge a suggestion from serving in minutes (the kill switch, again). Raising this unprompted is a strong product-maturity signal.

    Personalization, scoped honestly: full per-user ranking re-breaks the caching that makes the design work. The pragmatic layer: global top-K from the shared structure + a client-side/user-service re-rank over that small candidate set (your recent searches float up) — candidate generation vs. ranking, the feed's lesson, recursed. Say the trade: personalization depth costs cache hit rate, and the budget picks the winner.

    DevOps lens

    The operational story is snapshot-shaped: build-pipeline health is the freshness SLO (staleness alert on snapshot age — a wedged build means suggestions quietly fossilize; the dead-man's-switch pattern applies), swap safety (each snapshot canaried against golden queries — "empty results for 'a'" must fail the build, not reach users; rollback = re-point at the previous snapshot, seconds), serving-tier flatness (per-node p99 and hot-prefix skew — the L1 layer should flatten it; if one shard runs hot, a short prefix leaked past the edge cache), and QPS hygiene at the front door (per-client rate limits because scrapers love typeahead endpoints — they're a free query-popularity API if you don't meter them, which is also a competitive-intelligence leak worth one security sentence).

    Staff+ altitude

    (1) Typeahead is a product surface with editorial liability — what you suggest is what you endorse: the blocklist governance (who decides, how fast, audit trail) is a policy system wearing an infra costume, and the Staff move is designing the process alongside the pipeline. (2) The snapshot pattern is a platform asset — build-verify-swap-rollback over immutable artifacts generalizes to search indexes, ML feature stores, and config planes; building it once as shared machinery is the paved-road play. (3) Freshness tiers by query class — head queries (stable, cache-forever) vs. trending tail (streaming lane) lets you buy breakout-freshness without re-architecting the whole pipeline — the asymmetric-workload split, third appearance. (4) Measure the product, not just the service: suggestion acceptance rate (did the user click one?) is the metric that justifies the entire system's existence — and connects the infra conversation to the business one, which is the altitude shift in miniature.

    In the interview

    The spine: keystroke-QPS math → "precompute top-K per prefix, hash map over trie for ops simplicity" → the edge/client caching stack → the refresh pipeline with atomic snapshot swap → blocklist + rate limits as the maturity close. Probes to expect: why not compute at query time? (millions of descendants under short prefixes — the budget dies; precompute moves the work to write time where all read-optimization lives); how fresh are trends? (pipeline cadence + the streaming lane for breakouts — and ask what the product needs before gold-plating); how do you handle typos? (park it honestly: fuzzy matching is a different structure — edit-distance candidates or n-grams — layered behind the exact path so the hot path stays O(1)); what breaks at 10×? (nothing interesting — the dataset barely grows; QPS scales with stateless serving + more edge cache; the pipeline is the only tier that notices). The memorable close: "this is the purest read-path system in the canon — everything hard happens before the user types."
