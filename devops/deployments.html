    Deployment Strategies¶

    Here's the statistic that reorganizes priorities: most outages are self-inflicted, and most of those ship in a change. Which means deployment strategy isn't release-engineering trivia — it's the availability lever with the highest ROI in the building: making changes gradual, observable, and reversible attacks the dominant failure cause directly. This page is the strategy menu, the feature-flag layer that upgrades all of them, and the schema-migration choreography that makes "just roll back" true instead of aspirational.

    The strategy menu¶

    Rolling update — replace instances a few at a time behind the LB/Service. The default (capacity-efficient, no extra fleet), with two structural caveats: two versions run simultaneously during the roll (N and N+1 must coexist — the compatibility constraint that rules everything below), and rollback is a re-roll (minutes, not instant). The zero-downtime pod choreography is table stakes underneath.

    Blue-green — two full environments; flip traffic atomically at the router; rollback is flipping back. Instant reversibility, no version coexistence in traffic — priced at 2× fleet during the window, and the flip is all-or-nothing: 100% of users meet the bug at once, which is exactly the blast-radius philosophy this page exists to avoid. Best for: small fleets, protocol/runtime swaps where coexistence is genuinely impossible, and databases' cousins (the expand-contract alternative usually beats it there).

    Canary — the strategy the industry converged on: ship to a tiny slice (1%, one AZ, internal users first), compare against control on real metrics (error rate, p99, business KPIs — canary analysis, automated where mature: statistical comparison against the baseline fleet, auto-rollback on regression), then widen in waves with soak time between (1% → 5% → 25% → 100%, hours-to-days for risky changes). Canary converts deploys from bets into experiments with bounded downside — the staged-rollout answer to correlated failure. Requirements people forget: a traffic-splitting layer, comparable cohorts (same traffic mix in canary and control, or your analysis is astrology), and enough traffic per wave for statistical signal.

    Progressive multi-region — the fleet-scale wrapper: region by region, AZ by AZ, never all at once, with bake time proportional to blast radius. The discipline that makes "redundancy" survive its own deploy pipeline — because simultaneous global rollout is the correlated failure you built three regions to avoid.

    Feature flags: decoupling deploy from release¶

    The upgrade that changes the game: deploy (code reaches production, dark) and release (users experience it) become independent operations. Ship continuously; release deliberately — per user, per tenant, per percentage — and unrelease in milliseconds without touching infrastructure: the flag flip is the fastest mitigation in existence, no diagnosis required.

    The operational fine print that separates practitioners: flags are config with fleet-wide blast radius (canary flag changes like code — the biggest outages of the flag era were bad flag pushes, not bad code); the flag service must fail static (cached last-known values, safe defaults — a flag-service outage that takes your product down means you built a tier-0 dependency without noticing); and flags are inventory with decay — every long-lived flag doubles your test matrix (2ⁿ combinations at n flags), so expiry dates and cleanup sprints are hygiene, not perfectionism. (Design one properly in the case study.)

    The database is the hard part¶

    Code rolls back; data doesn't. Every strategy above silently assumes the schema works for both running versions — which is exactly the expand–contract discipline: additive-only changes (expand), dual-write/backfill behind the scenes, flip reads via flag, contract later — never a destructive change in the same deploy as the code that stops needing the old shape. The compatibility matrix to keep in your head: version N+1 must run against schema S and S+1 (deploy order freedom), and version N must survive schema S+1 (rollback freedom). Break the second and your "instant rollback" is a fiction — the moment old code meets new data and chokes, you're fixing forward under pressure, which is the failure mode this entire page exists to prevent. The same coexistence law governs API/message contracts between services: additive evolution isn't politeness, it's what makes independent deployment possible.

    Rollback as a designed capability¶

    "We can roll back" is a claim with requirements: the previous artifact still exists and is one promotion away; config and flags version with the code (rolling back code under new config is a third, untested state); data compatibility holds (above); and the procedure is rehearsed and fast — rollback-time is MTTR's biggest component, and a 40-minute rollback ceremony converts blips into incidents. The maturity ladder: manual rollback → one-click → automated on canary regression — and the cultural rule that makes it work: rollback is the default response to deploy-correlated regression; diagnosis happens afterward, on a healthy system (mitigate first, institutionalized).

    DevOps lens

    Deploy observability is its own discipline: annotate everything with deploy markers (the vertical line on every dashboard that answers "what changed?" — the first question of every incident), watch deploy-correlated error/latency deltas rather than absolute thresholds (a 0.3% error bump invisible to alerting is a screaming canary signal when correlated with the rollout wave), and instrument the pipeline's own funnel (deploys started/completed/rolled-back, time-in-wave — change-failure rate and MTTR fall out of this). The incident genres: the slow-burn regression (memory leak that fires two days post-deploy — long soak times and working-set trend alerts catch it; this is why "deploy Friday?" is really "who watches the soak?"), the config-code skew (rollback restored the binary but not the flag state — version them together), and the mixed-fleet mystery (bug only on the 30% new-version fleet: bimodal latency, confusing every average — segment metrics by version, always).

    Staff+ altitude

    Markers: (1) Deployment policy is availability policy — wave sizes, soak times, freeze windows, and auto-rollback thresholds tiered by service criticality, written as platform doctrine with rationale (error-budget-linked: burn the budget, slow the rollouts — the SRE contract made mechanical). (2) The compatibility law is a governance artifact — additive-only schema/API evolution enforced in CI (linters on migrations and proto diffs), because the law only works if everyone obeys it always; one team's breaking change is every team's rollback trap. (3) Flag governance before flag sprawl — ownership, expiry, blast-radius tiers, and change-canarying for the flag system itself; the alternative is 4,000 flags of unknown state, which is configuration drift with a UI. (4) Measure the metabolism — deploy frequency and change-failure rate per team, used diagnostically, not punitively: teams that ship small and often have better availability because of it, and proving that with your own org's data is how a Staff engineer ends the "deploys are risky, slow them down" debate permanently.

    In the interview

    The rollout move — narrating deployment unprompted — is your signature; this page is its full ammunition: "Ship behind a flag, dark. Canary 1% in one AZ with automated analysis against control; auto-rollback on regression. Widen in waves with soak time, region by region. Schema changes ran expand-contract two deploys ahead, so rollback is one promotion with no data hazard. The flag is the millisecond kill switch; the canary is the blast-radius cap." Probes to expect: "blue-green vs. canary?" (instant-flip reversibility vs. bounded-blast-radius experimentation — and canary won the industry because correlated failure is the enemy); "how do you roll back a schema change?" (you don't — you design so you never have to: expand-contract, compatibility matrix, contract-later); "two versions running at once — problems?" (the coexistence law: wire/schema compatibility, plus version-segmented metrics for the mixed-fleet mystery); "when would you NOT canary?" (sub-statistical traffic, true atomicity needs — then blue-green with a rehearsed flip). The through-line to make explicit: deploys are the dominant outage cause, so deployment design is reliability design — that framing alone is a Staff-level opening sentence.
