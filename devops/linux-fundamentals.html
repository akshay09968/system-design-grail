    Linux Fundamentals for Scale¶

    Every abstraction in this site — containers, pods, meshes, serverless — eventually leaks, and what it leaks down to is a Linux box: processes, file descriptors, pages of memory, a scheduler, and a network stack. The engineers who debug fastest and design most credibly are the ones who can see through the abstraction to the substrate. This page is the substrate, selected for what actually matters at scale — which is to say, selected by what has actually caused outages.

    Processes, threads, and what the scheduler owes you¶

    A process owns memory and file descriptors; threads share their process's memory and get scheduled independently. The kernel's CFS scheduler time-slices runnable threads across cores — and the two numbers that matter operationally are load average (runnable + uninterruptible-wait tasks; load 64 on a 64-core box is full, not dying — normalize by cores) and run-queue latency (how long runnable threads wait for a core — the invisible tax that appears in your p99 before it appears anywhere else).

    The scale-relevant facts: context switches cost microseconds and cache pollution — thousands per second per core is fine, hundreds of thousands is a design smell (too many chatty threads; see the thread-per-connection lesson); a process that can't get scheduled looks exactly like a slow process to its callers (gray failure origin story); and CPU throttling under cgroup quotas (the container page's sharp edge) means a "healthy" pod can be paused dozens of times a second — the p99 spike with no cause in any application log.

    File descriptors: everything is one, and they run out¶

    Sockets, files, pipes, epoll instances, timers — in Linux, everything is a file descriptor, and FDs are a bounded per-process resource (ulimit -n). The outage genre: service healthy, CPU idle, and every new connection failing with EMFILE — because a leak (connections not closed on error paths, the classic) exhausted the table. FD count is a first-class metric; a steady upward drift on a steady-traffic service is a leak with a countdown attached.

    The event-loop story (epoll, C10K) lives here too — one thread monitoring 100k FDs is why modern proxies and runtimes scale; the substrate view adds the caveats: accept() backlogs (SYN queue overflow drops connections silently), TIME_WAIT accumulation on churning clients, and conntrack limits on NAT'd paths — the four classic network incidents, all of them ultimately FD-and-kernel-table stories.

    Memory: the beautiful lies¶

    Linux memory is a stack of productive deceptions worth seeing through:

    • Virtual memory & overcommit — processes are promised address space that isn't backed until touched; the kernel happily promises more than exists. The bill collector is the OOM killer: when physical memory truly runs out, the kernel selects a process and kills it (badness score ≈ memory used, adjustable). In containers, cgroup memory limits scope this: exceed your limit and your container gets OOMKilled (exit 137 — the most-Googled exit code in Kubernetes history). Design consequence: memory limits are a contract with a death penalty; set them from measured RSS + headroom, not vibes.
    • The page cache is where your "free" memory went — Linux uses idle RAM to cache file pages, and this is the performance substrate of databases and Kafka ("free" memory near zero is healthy; available is the number that matters). The design corollary: co-locating a page-cache-hungry workload with a memory-ballooning neighbor evicts the cache and degrades the first workload with no metric pointing at the culprit — noisy neighbors at the kernel layer.
    • Swap in production — mostly disabled (Kubernetes historically required it off): better a fast OOM kill than a service running at disk speed pretending to be alive (fail-stop beats gray failure, as policy).
    • Huge pages, NUMA — one line each: TLB pressure matters for giant-memory databases (transparent huge pages: measure, sometimes disable); cross-NUMA memory access costs ~1.5–2× — pinning matters at the extreme end (and explains "same hardware, different latency" mysteries on big boxes).

    cgroups & namespaces: the primitives under everything¶

    Preview of the containers page, from the kernel's side: cgroups meter and limit resources (CPU quota/shares, memory ceilings, IO weight, PIDs) per process-tree; namespaces scope visibility (PID, network, mount, user...). A container is nothing more than a process launched inside both. What matters here: cgroups are also how systemd organizes everything on a modern box, how Kubernetes enforces requests/limits, and how you can meter any process-tree on any Linux box without a container in sight — systemd-run --scope -p MemoryMax=2G ./job is a rite-of-passage trick.

    The kernel's dials that page you¶

    A short list, selected by incident frequency: somaxconn/SYN backlog (accept-queue overflow under connection floods — silent drops), conntrack table size (NAT/firewall flow tracking; full = new connections vanish, the most confusing incident shape), ephemeral port range (outbound-heavy proxies exhaust it — TIME_WAIT reuse), fs.file-max and per-process ulimits, TCP buffer sizes (long-fat-pipe throughput — bandwidth-delay product), and dirty-page writeback thresholds (the mystery pause when a write-heavy process forces synchronous flush — "the box froze for 4 seconds" has this as a standing suspect). You don't memorize values; you know the names and failure signatures, which is what turns a 3-hour mystery into a 10-minute fix.

    Observability at the substrate¶

    The USE method (utilization, saturation, errors — per resource) is the Linux-layer complement to service-level RED metrics: CPU (run-queue depth beats %util), memory (available + major page faults + OOM kills), disk (IO latency and queue depth — a 100%-util NVMe at 100 µs is fine; a 60%-util disk at 40 ms is dying), network (drops, retransmits, conntrack fill). The modern deep-inspection layer is eBPF — safe kernel-side programs powering the current generation of profilers, network observability, and security tooling (the mesh-without-sidecars trend); you don't need to write it, but "we'd get syscall-level visibility via eBPF without touching app code" is a real answer to real questions now.

    DevOps lens

    This page is the ops lens, so the callout goes to the incident-recognition table: p99 spikes, app logs silent → CPU throttling (cgroup quota) or run-queue saturation or dirty-page flush — check throttled_time, run-queue, writeback, in that order. Exit 137 / random pod deaths → OOM: limit vs. actual RSS (and check for the memory-ballooning neighbor evicting page cache). New connections fail, established ones fine → FD exhaustion, backlog overflow, or conntrack full — three tables, three checks. "The box froze" → writeback storm, swap thrash (if someone left swap on), or hypervisor steal time (check %st — your VM paused, not your code). Every one of these is a five-minute diagnosis if you know the shape, and a five-hour mystery if you're only looking at application dashboards.

    Staff+ altitude

    Markers: (1) Capacity models start at the substrate — "how many pods per node" is really FDs + memory-with-page-cache-honesty + conntrack + CPU-throttling headroom; Staff-level capacity plans cite kernel limits, not just CPU averages. (2) Own the kernel-settings paved road — sysctl profiles, ulimits, and cgroup defaults belong in the base image/node config as reviewed artifacts (the alternative is every team rediscovering somaxconn during their own outage). (3) Know when to descend — most engineering shouldn't touch this layer, but Staff judgment includes recognizing the problems that only resolve here (the 40 ms Nagle mystery, the NUMA latency split, the writeback pause) and descending fast instead of adding another cache on top of a substrate problem. (4) The meta-skill this page trains is seeing through abstractions — the same move as reading Kafka as page cache or K8s limits as cgroups; abstractions are for building, substrates are for debugging.

    In the interview

    This content surfaces as depth flexes attached to other answers: sizing a connection-heavy service → mention FD limits and epoll ("a node holds 100k+ idle WebSockets fine — the constraints are FDs and memory, not CPU"); sizing memory → mention page cache honestly ("the database wants RAM for page cache, so 'free' memory isn't free capacity"); explaining a latency mystery the interviewer poses → the throttling/steal-time/writeback trio shows diagnostic range. And one full set piece worth having: what happens when a container exceeds its memory limit — cgroup ceiling → OOM killer selects within the cgroup → exit 137 → kubelet restarts per policy → and the design response (limits from measured RSS + headroom, alerts on working-set-vs-limit ratio, and load-shedding before allocating in the app itself). It's thirty seconds of pure operational credibility.
