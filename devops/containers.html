    Containers Deep Dive¶

    The most clarifying sentence in all of container literature: a container is just a Linux process — launched with its visibility scoped by namespaces, its resources metered by cgroups, and its filesystem assembled from stacked tar archives. No hypervisor, no guest kernel, no magic. Everything about containers — the millisecond starts, the density, the security model's limits, the image ecosystem — falls out of that one sentence, and interviewers probing "what is a container?" are checking whether you know it or only know Docker commands.

    The three ingredients¶

    Namespaces scope what a process can see. PID namespace: the process is PID 1 of its own little world, blind to the host's process table. Network namespace: its own interfaces, routes, ports (this is why every pod gets its own IP — a fresh network namespace with a veth pair). Mount, UTS, IPC, user namespaces: its own filesystem view, hostname, shared-memory world, and UID mapping (user namespaces map container-root to an unprivileged host UID — the security upgrade that took the industry embarrassingly long to make default).

    cgroups scope what a process can use. The kernel's metering: CPU quota (with the throttling tax attached), memory ceiling (with the OOM killer as enforcement), IO weight, PID counts. Kubernetes requests and limits compile down to exactly these knobs — there is no other enforcement layer.

    Layered images scope what a process starts from. An image is an ordered stack of read-only tarballs (layers) plus metadata; the runtime unions them (overlayfs) and adds one writable layer on top. The consequences do real work: layers dedupe (fifty services on one base image store the base once — density economics), cache (rebuild only the layers whose inputs changed — CI speed lives and dies here: order your Dockerfile from stable to volatile, dependencies before code), and are content-addressed (a layer's digest is its identity — the CAS pattern again, buying free integrity and registry dedupe).

    The sharpest contrast to have ready: containers share the host kernel. A VM boots its own; a container is a scoped view of yours. That's why containers start in milliseconds and pack densely — and why container isolation is kernel-enforced, not hardware-enforced: a kernel exploit escapes every container on the node. For hostile multi-tenancy, the industry answers with micro-VMs (Firecracker — Lambda's substrate) and sandboxed runtimes (gVisor): VM-grade isolation at near-container speed. "Namespaces for teams, micro-VMs for strangers" is the one-line tenancy policy.

    Images done well (the craft section)¶

    • Small images are a security and ops feature: less attack surface, faster pulls (pull time is cold-start time — it gates scale-out speed and deploy speed), less to scan. Multi-stage builds (build in a fat image, copy artifacts into a distroless/minimal runtime image) are the standard move; "our runtime image has no shell" is a sentence security teams love.
    • Immutability + digest pinning: tags are mutable pointers (:latest is a bug report waiting for a timestamp); production references images by digest, and the same digest flows dev → staging → prod (build once, promote everywhere — the artifact is the invariant; only config varies).
    • The registry is tier-0 infrastructure: every scale-out event and node replacement pulls from it; registry down = cluster can't heal or grow. Treat it accordingly — replicated, cached-per-region/cluster (pull-through caches), rate-limit-aware (the Docker Hub limits incident of 2020 taught everyone this the hard way).
    • Supply chain, one paragraph of literacy: sign images (Sigstore/cosign), generate SBOMs, scan in CI, and admit only signed images at the cluster boundary (admission controllers) — the deploy pipeline is your real perimeter, and image provenance is its passport system.

    Runtime realities¶

    The stack under Kubernetes: kubelet → CRI (containerd, CRI-O) → OCI runtime (runc) — Docker-the-daemon left the building years ago; Docker-the-format (OCI images) is the universal standard. Operational edges worth owning: PID 1 duties (your process is init inside its namespace — it must reap zombies and forward signals; tini or proper handlers, or SIGTERM never reaches your app and graceful shutdown silently becomes SIGKILL), writable-layer discipline (writes to the container filesystem are ephemeral and slow-ish; real data goes to volumes — state doesn't live in photocopies), and one process-tree per container (compose systems from containers-per-concern, not systemd-in-a-container).

    DevOps lens

    The container incident canon: image-pull failures as availability events (registry outage/rate limit during a scale-out or node cycling = capacity frozen; pull-through caches and pre-pulled critical images are the fix), the :latest drift mystery (two nodes, two different "same" images — digest pinning ends the genre), zombie/signal bugs (deploys that take 30 s of SIGKILL instead of 2 s of graceful drain — check who's PID 1), disk pressure from image/layer accumulation (kubelet GC tuning; the node that's "out of disk" while running three pods), and build-cache poisoning of CI times (one layer-order refactor turning 90-second builds into 20-minute builds fleet-wide). None of these are exotic; all of them have paged someone you know.

    Staff+ altitude

    Markers: (1) Base images are a product — a platform team ships and patches a small set of blessed bases (with SLAs on CVE turnaround), because 400 teams choosing bases independently is 400 patch programs and one shared incident when the unpatched one meets a scanner — or an attacker. (2) Tenancy isolation is a written spectrum — same-team workloads share nodes; different-trust workloads get node pools or micro-VMs; customer code gets Firecracker-class isolation, full stop; the Staff artifact is the policy mapping trust level → isolation tier, decided before the incident. (3) Registry topology is availability architecture — per-region registries/caches, replication, and "can the fleet self-heal with the origin registry down?" belong in DR planning alongside databases. (4) Provenance as a gate, not a report — signing + admission policy converts supply-chain security from quarterly-scan theater into a structural invariant; it's the constraint-as-last-line philosophy applied to artifacts.

    In the interview

    The set piece: "what is a container, actually?" → the one-sentence answer (namespaces for visibility, cgroups for resources, layered images for filesystem — a scoped process, not a VM), then one consequence per ingredient (pod-per-IP from netns; limits enforced as cgroup/OOM from cgroups; build caching and CAS from layers). Follow-ups: "container vs. VM?" (shared kernel — millisecond starts and density vs. hardware-grade isolation; name the micro-VM middle ground for multi-tenant designs); "why did the pod die with exit 137?" (the OOM walk); "how do you make deploys/scale-outs fast?" (image size + layer order + registry proximity — pull time is cold-start time). In any design with untrusted code execution (CI runners, plugin platforms, serverless), the isolation-tier answer — "containers for our code, Firecracker for theirs" — is the difference between a hand-wave and a design.
